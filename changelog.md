All notable changes to this project are documented in this file. Dates are in ISO 8601 format (UTC).

## [Unreleased]

- Install directory moved from `C:\ProgramData\DNSGuard` to `C:\ProgramData\OSGuard`.
- Task names renamed from `DNS-Hijack-Guard` to `OS-Guard-Protection` and associated guardians.
- Added `ChildLogon` scheduled task that applies HKCU restrictions directly in the child's session at every logon.
- Added `Mount-ChildHive` / `Dismount-ChildHive` helper functions to load and edit the child's `NTUSER.DAT` offline.
- `New-LocalUser -NoPassword` used for passwordless child account creation with `net user /passwordchg:no /passwordreq:no` hardening.
- Added `Enable-OSLock` and `Disable-OSLock` functions alongside preserved `Enable-DNSLock` / `Disable-DNSLock`.
- Interactive menu now shows DNS status, OS child lockdown status, and installation integrity in a single unified dashboard.
- WMI event subscription renamed to `OSGuardWmiHealth`.
- Integrity hash registry key renamed from `PushConfigBackoffInterval` to `OSGuardIntegrity`.
- Added regression test suite in `tests/` folder with `syntax_check.ps1` and `test_os_lockdown.ps1`.

- **Category Status Grid Bug Fix** (2026-06-29T21:28:00Z): Fixed `NullArrayIndex` runtime error in `Show-CategoryGrid` (`new2_OS_lockdown.ps1:2591`). Changed `$Keys = $Categories.Keys` to `$Keys = @($Categories.Keys)` to cast the keys into a standard array before the two-column grid loop. This prevents the grid from crashing when rendering lock categories in the TUI.

- **Category Grid [UNKNOWN] Fix** (2026-06-30T12:40:00Z): Fixed `Show-CategoryGrid` rendering `[UNKNOWN]` for all child-hive categories when the child account has never logged in (no `NTUSER.DAT` exists). - **Temporary Admin Unlock (Option 2 Toggle)** (2026-06-30T13:15:00Z): Menu option `[2]` is now a toggle between `TEMPORARY UNLOCK` and `RE-LOCK SYSTEM`. When pressed, it unlocks machine-wide policies (DNS + OS) while preserving child account restrictions, sets the `OSGuardTempUnlockActive` registry flag, and suppresses the 5/10-minute guardian tasks from re-locking. The `OSGuard-ParentModeWatch` AFK watcher was extended to monitor both `OSGuardParentModeActive` and `OSGuardTempUnlockActive`; after 5 minutes of idle time it auto-triggers `oslock -LockNow`. Pressing `[2]` again (or `[8] LOCK NOW`) manually clears the flag and re-locks. `Exit-ParentMode` and `Uninstall-Persistence` also clear the temp unlock flag. `Enable-OSLock` clears the flag on entry to prevent stale state.
- **Temp Unlock Countdown Banner + AFK 1 Minute** (2026-06-30T13:29:00Z): The interactive menu now displays a prominent `>>> TEMP UNLOCK ACTIVE - Countdown: mm:ss remaining (auto-locks after 1 min idle) <<<` banner when the temporary unlock is active. The banner computes the elapsed time since `OSGuardTempUnlockTimestamp` and shows an approximate countdown against the 1-minute AFK threshold. The `OSGuard-ParentModeWatch` embedded Base64 script was updated to reduce the idle timeout from 5 minutes to 1 minute (`$Timeout = 1 * 60 * 1000`) so both Parent Mode and temporary unlock auto-lock faster. The post-unlock confirmation message also reflects the new 1-minute threshold.
- **Live Temp Unlock Countdown Window** (2026-06-30T13:33:00Z): A dedicated live countdown window (`OS-Guard Temp Unlock Timer`) pops up when temporary unlock is activated. It uses a `System.Windows.Forms.Timer` to update every second, showing the remaining time in `mm:ss` format. The window stays `TopMost`, is fixed-size (420x160), and centered on screen. If the user closes the window via the X button, the `FormClosing` event triggers `oslock -LockNow` immediately. If the countdown reaches zero, it also auto-triggers `oslock -LockNow`. The timer process writes its PID to `$InstallDir\TempUnlockTimer.pid` so the main script can kill it when the admin re-locks manually via option `[2]` or `[8]`. The timer script is written to `$InstallDir\TempUnlockTimer.ps1` during install and reused. New helper functions `New-TempUnlockTimerScript`, `Start-TempUnlockTimer`, and `Stop-TempUnlockTimer` manage the lifecycle. `Enable-OSLock`, `Exit-ParentMode`, and `Uninstall-Persistence` all call `Stop-TempUnlockTimer` to prevent stale windows.
- **Temp Unlock Timer Directory Fix** (2026-06-30T13:44:00Z): Fixed `New-TempUnlockTimerScript` and `Start-TempUnlockTimer` crashing with `Could not find a part of the path 'C:\ProgramData\OSGuard\TempUnlockTimer.ps1'` when the script was run without being installed (i.e., `$InstallDir` does not exist). Both functions now create `$InstallDir` automatically via `New-Item -ItemType Directory -Force` before writing the timer script or PID file, so the temporary unlock works even in standalone mode.
- **Standalone Mode File Write Fixes** (2026-06-30T13:45:00Z): Fixed `Export-OSGuardReport` crashing with `Could not find a part of the path` when run without installation because `Export-Csv` does not create parent directories. Added `New-Item -ItemType Directory` before the export with a `try/catch` wrapper. Also fixed `Write-Log` silently failing when `$InstallDir` does not exist; it now auto-creates the directory before appending to the log file so standalone-mode runs still produce logs. Scanned all other `Set-Content` / `Out-File` / `Export-Csv` sites in the script; `New-BrowserLauncher`, `Start-WindowGuard`, `Set-ParentPassword`, and `SilentLock` already have `try/catch` blocks that prevent crashes, but the new `Write-Log` directory creation ensures consistent logging across all paths.
|- **Fresh Install Guard Fix** (2026-06-30T19:41:00Z): Changed `Install-Persistence` installation gate and menu option `[3]` guard from checking `$InstallDir` to checking `$InstallScript`. This prevents installation from being blocked when the directory was auto-created by a previous `Write-Log` run but the actual payload `OS_Lockdown.ps1` is missing (e.g., after a Windows reinstall or manual directory cleanup). The guard now verifies the installed script file exists rather than merely the directory, and `Uninstall-Persistence` correctly removes the directory afterward.
|- **Auto-Enable ExecutionPolicy** (2026-06-30T19:45:00Z): When the script detects it is running elevated, it automatically checks the current user's `ExecutionPolicy`. If it is `Restricted`, `AllSigned`, or `Undefined`, the script sets `Set-ExecutionPolicy RemoteSigned -Scope CurrentUser -Force` so the script (and future PowerShell runs) work without requiring the user to manually change execution policy. A log entry is written on success or failure.
|- **Apply-EdgePolicies Get-Member Fix** (2026-06-30T19:50:00Z): Fixed `Get-Member : You must specify an object` error when the `URLBlocklist` registry key exists but has no properties yet. The pipeline `Get-ItemProperty ... | Get-Member ...` was receiving `$null` under `Set-StrictMode -Version Latest` and throwing. Split the pipeline to capture the property object first and only call `Get-Member` when the object is not `$null`.
|- **Uninstall SYSTEM Cleanup Fix** (2026-06-30T19:52:00Z): Fixed `Uninstall-Persistence` incorrectly reporting failure because `Invoke-AsSystem` was calling `Write-Log` after the SYSTEM task had already deleted `$InstallDir`. `Write-Log` auto-creates the directory if it is missing, so `Test-Path` immediately afterward found the directory and reported a failure. Changed `Invoke-AsSystem` to use `Write-Host` for its internal debug messages instead of `Write-Log`, and added a fallback direct `Remove-Item` after the SYSTEM task returns so ACLs reset by the task can be used for a final cleanup attempt.
|- **Write-Log Directory Recreate Fix** (2026-07-01T18:39:00Z): Fixed the root cause of the uninstaller falsely reporting `Install directory still exists`. `Write-Log` was auto-creating `$InstallDir` whenever it was missing, which meant that after `Uninstall-Persistence` deleted the directory, any subsequent `Write-Log` call silently recreated it. Added a `$script:SuppressLogDirCreation` flag that is set to `$true` at the start of `Uninstall-Persistence` and reset to `$false` after the final verification, preventing `Write-Log` from recreating the directory during uninstall. Also hardened `Write-Log` to only write to the log file if the directory actually exists.
|- **Memorable Password Generation** (2026-07-01T19:00:00Z): Replaced the random 12-character alphanumeric default password with a human-memorable passphrase generator (`New-MemorablePassword`). Passwords now use a common English word + 2-digit number + an easy symbol from a reduced keyboard-safe set (`! # $ % & + - = ? _ @`), assembled in a random order pattern (e.g., `Dragon42!`, `!42Dragon`, `42!Dragon`). `Set-ParentPassword` displays three live suggestions before prompting. Minimum length reduced from 8 to 6 characters with relaxed complexity rules (`Test-PasswordComplexity`: requires at least one letter and one number). The default install password, tamper lockout screen, and window guard all use the new memorable format.
